# 米家控制面板重构计划

## 需求概述

### 需求1：设备配置统一存储

* **现状**：设备列表存储在两处

  * `DEFAULT_DEVICES` 变量（app.py 第28-60行）：硬编码的设备列表

  * `~/.mijia-panel/devices_config.json`：动态配置文件

* **目标**：将所有设备统一存储在配置文件中，移除代码中的硬编码

### 需求2：智能开关设备聚合展示

* **现状**：Mesh开关设备（mesh\_switch）按独立设备展示

  * 同一物理设备（相同did）的不同按键（不同siid）作为独立设备

  * 例如：`did=1132827761` 有 siid=2,3,4,12 四个独立设备

* **目标**：前端聚合展示

  * 同一did的设备聚合为一个智能设备

  * 每个siid作为子开关显示在设备内部

***

## 实施步骤

### 第一阶段：设备配置迁移（后端）

#### 1.1 创建配置迁移函数

* 新增 `migrate_default_devices()` 函数

* 检查配置文件是否已有设备数据

* 首次运行时将 DEFAULT\_DEVICES 写入配置文件

* 迁移成功后标记配置文件避免重复迁移

#### 1.2 修改配置加载逻辑

* `load_config()`: 移除 DEFAULT\_DEVICES 作为默认值的逻辑

* `get_all_devices()`: 直接从配置文件读取，配置文件为空时返回空列表

* `get_display_devices()`: 保持现有逻辑，从配置文件读取

#### 1.3 保留 DEFAULT\_DEVICES 作为迁移数据源

* 暂时保留 DEFAULT\_DEVICES 变量用于迁移

* 添加注释说明该变量仅用于首次迁移

* 后续版本可移除

#### 1.4 更新全局变量初始化

* `DEVICE_MAP` 需要在配置加载后动态更新

* 确保服务启动时正确初始化

***

### 第二阶段：Mesh开关聚合逻辑（后端）

#### 2.1 新增数据结构

```python
# 聚合后的Mesh设备结构
{
    "id": "mesh_1132827761",           # 聚合设备ID
    "name": "客厅开关",                  # 设备名称（取第一个子设备名称或自定义）
    "type": "mesh_switch_group",        # 新类型标识
    "gateway_id": "gateway_main",       # 网关ID
    "did": "1132827761",                # 设备DID
    "sub_switches": [                   # 子开关列表
        {
            "id": "mesh_switch_living_lamp",
            "name": "客厅灯",
            "siid": 2,
            "power": true
        },
        ...
    ],
    "online": true,
    "controllable": true
}
```

#### 2.2 新增聚合函数

* `aggregate_mesh_switches(devices)`: 将mesh\_switch设备按did聚合

* 保留原始设备数据，聚合仅用于展示层

#### 2.3 修改API响应

* `/api/status/all`: 返回聚合后的mesh设备状态

* `/api/status/<device_id>`: 支持查询子开关状态

* `/api/control/<device_id>`: 保持现有控制逻辑不变

#### 2.4 新增API端点（可选）

* `/api/mesh/aggregate`: 返回聚合后的mesh设备列表

## 风险评估

### 低风险

* 配置迁移：仅首次运行执行，有幂等性保护

* 前端展示修改：不影响后端数据结构

### 中风险

* Mesh聚合逻辑：需要确保控制功能不受影响

* 建议：保留原始设备数据，聚合仅用于展示层

### 缓解措施

* 添加配置备份机制

* 保留原始设备数据结构

* 充分测试各种场景
