## 一、WiFi 直连设备

**特点**：设备直接连接家庭WiFi，有独立IP地址

**设备类型**：灯光、空调、加湿器、插座、传感器、音响、家电

**通信方式**：

```python
# 设备配置示例
{"id": "light_living", "name": "护眼客厅吸顶灯", "ip": "192.168.3.19", "token": "xxx", "type": "light"}
```

**查询流程**：

```
后端 ──────► 设备IP:192.168.3.19 ──────► 设备响应状态
      miio协议
```

```python
# 代码实现
dev = Device(ip="192.168.3.19", token="xxx", timeout=2)
props = dev.send("get_properties", [{"siid": 2, "piid": 1, "did": "light_living"}])
# siid: 服务ID (Service ID)
# piid: 属性ID (Property ID)  
# did:  设备ID
```

**控制流程**：

```python
dev.send("set_properties", [{"siid": 2, "piid": 1, "value": True, "did": "light_living"}])
```

## 二、蓝牙 Mesh 设备

**特点**：设备通过蓝牙Mesh网络连接，需要通过网关中转通信

**设备类型**：mesh_switch（智能开关）

**设备配置**：

```python
# Mesh设备配置 - 没有独立IP，通过网关通信
{
    "id": "mesh_switch_living_lamp", 
    "name": "客厅灯", 
    "gateway_id": "gateway_main",  # 关联的网关
    "did": "1132827761",           # Mesh设备ID
    "siid": 2,                      # 服务ID（同一物理开关的不同按键用不同siid）
    "type": "mesh_switch"
}

# 网关配置 - 有独立IP
{
    "id": "gateway_main", 
    "name": "Xiaomi Home Hub", 
    "ip": "192.168.3.77", 
    "token": "xxx", 
    "type": "gateway"
}
```

**通信方式**：

```
后端 ──────► 网关IP:192.168.3.77 ──────► 蓝牙Mesh网络 ──────► Mesh设备
      miio协议                              蓝牙BLE
```

**查询流程**：

```python
# 通过网关查询Mesh设备
gateway = Device(ip="192.168.3.77", token="xxx", timeout=3)
props = gateway.send("get_properties", [
    {"siid": 2, "piid": 1, "did": "1132827761"}  # did是Mesh设备ID
])
```

**控制流程**：

```python
gateway.send("set_properties", [
    {"siid": 2, "piid": 1, "value": True, "did": "1132827761"}
])
```

## 三、关键区别对比

| 特性 | WiFi 直连设备 | 蓝牙 Mesh 设备 |
|------|--------------|---------------|
| **IP地址** | 有独立IP | 无，通过网关IP |
| **通信路径** | 后端 → 设备 | 后端 → 网关 → 设备 |
| **Token** | 设备自己的Token | 使用网关的Token |
| **并发查询** | 可并行查询所有设备 | 同一网关下需批量查询 |
| **响应速度** | 较快（1跳） | 较慢（2跳） |
| **典型设备** | 灯泡、空调、插座 | 智能开关按键 |

## 四、批量查询优化

**问题**：同一网关下的Mesh设备如果并行查询，网关会因并发过高而超时

**解决方案**：将同一网关的Mesh设备合并为一次批量查询

```python
# 批量查询 - 一次请求获取所有Mesh设备状态
props_list = [
    {"siid": 2, "piid": 1, "did": "1132827761"},  # 客厅灯
    {"siid": 3, "piid": 1, "did": "1132827761"},  # 餐厅灯
    {"siid": 2, "piid": 1, "did": "1133876907"},  # 主卧室灯
    # ... 更多设备
]
props = gateway.send("get_properties", props_list)
```

## 五、miio 协议核心概念

| 概念 | 全称 | 说明 |
|------|------|------|
| **siid** | Service ID | 服务ID，设备的不同功能模块 |
| **piid** | Property ID | 属性ID，服务下的具体属性 |
| **did** | Device ID | 设备标识 |
| **value** | - | 属性值（开关：true/false） |

**示例**：一个四键智能开关

```
物理设备 did: 1132827761
├── 按键1 (siid: 2) - 客厅灯
├── 按键2 (siid: 3) - 餐厅灯  
├── 按键3 (siid: 4) - 客厅开关按键3
└── 按键4 (siid: 12) - 客厅开关按键4
```

## 六、性能优化策略

### 6.1 后台缓存机制

- 服务启动时开启后台线程，每10秒自动刷新设备状态
- 前端请求 `/api/status/all` 直接返回缓存，响应时间 <10ms

### 6.2 并行查询

- WiFi设备：使用线程池并行查询
- Mesh设备：按网关分组，每组一次批量查询

### 6.3 异步控制

- 控制操作立即返回成功，后台线程执行实际指令
- 同时更新缓存状态，前端响应时间 <50ms

## 七、API 接口

| 接口 | 方法 | 说明 |
|------|------|------|
| `/api/status/all` | GET | 获取所有设备状态（返回缓存） |
| `/api/status/<device_id>` | GET | 获取单个设备状态（实时查询） |
| `/api/control/<device_id>` | POST | 控制设备开关 |
| `/api/devices/all` | GET | 获取所有设备列表 |
| `/api/devices/display` | POST | 设置设备显示/隐藏 |
